# 长线任务执行规范（工程版）

本文档用于约束“长线重构 / 审计 / 回归”类任务，目标是保证可回滚、可验证、可持续迭代。

## 1. 核心目标

1. 在不破坏现有功能的前提下，提升代码模块化、复用性与可维护性。
2. 优先消除重复实现、过深嵌套、跨文件耦合和不必要复杂度。
3. 每次变更都可独立回滚，并可通过自动化构建与页面回归验证。

## 2. 长线任务模板（优化措辞）

下面是一个长线任务。请持续自主迭代完成，不要等待额外指令。  
在整个执行周期内，按“审计 -> 小步重构 -> 验证 -> 本地提交锚定 -> 下一轮迭代”循环推进。  
若出现不确定项，优先通过代码、日志和测试定位；无法确认时暂停并提出问题。

必须完成的内容：

1. 审计 Astro 源码结构，减少冗余与过深嵌套，优先复用已有模块。
2. 审计 CSS 与 Tailwind，修复冲突与重复定义。
3. 审计 `scripts` 目录，合并可复用逻辑。
4. 审计 GitHub Workflows，修复安全与权限问题。
5. 先将可复用规则写入 `AGENT.md`，再开始重构。
6. 执行多轮自我迭代，直到结构清晰、复用充分。
7. 通过 Playwright 进行多页面回归，发现问题立即修复。
8. 对高风险尝试采用“先建锚点再改动”，必要时回滚重做。
9. 最终通过 `pnpm build` 与回归测试，并输出详细工作报告。
10. 每一轮迭代都要本地 git 提交，禁止自动推送远端。
11. 允许按需引入工具，但需说明目的与收益。
12. 始终保持项目可回退到任意稳定锚点。

## 3. 执行规则

1. 每轮改动完成后，立即本地提交，提交信息使用简体中文。
2. 除非用户明确要求，禁止推送远端。
3. 高风险改动前先创建锚点提交；失败后优先 `git revert` 或回到最近稳定提交重做。
4. 单轮只做一类改动（例如“脚本复用”或“工作流权限收敛”），避免混改。
5. 能抽公共模块就不复制粘贴；优先改造已有实现，不重复造轮子。
6. 每轮至少执行与改动匹配的验证（构建、类型检查、脚本自测或页面回归）。
7. 重构默认不改变对外行为；若行为变化，必须在报告中明确列出。

## 4. 质量门槛（DoD）

1. 代码结构更清晰：重复逻辑减少、模块职责边界更明确。
2. `pnpm build` 通过且无新增错误。
3. 关键页面在桌面端与移动端可正常访问与交互。
4. 每轮都有本地提交锚点，可独立回滚。
5. 最终报告包含：问题清单、改动说明、验证结果、残留风险、后续建议。

## 5. 安全与工作流基线

1. Workflow 权限遵循最小权限原则（默认 `read`，按需提升）。
2. 避免在高权限事件中暴露长期凭据，优先使用内置 `GITHUB_TOKEN`。
3. 外部网络请求必须设置超时与失败兜底，避免无界等待。
4. 自动化流程应配置并发与超时策略，防止资源占用失控。

## 6. 提交与发布规则（强约束）

1. 发布新文章时，提交信息必须严格使用：

`posts:发布新文章《XXX》。XXXXX。`

2. 每次更改都应作为本地 git 提交保存，便于回滚。
3. 推送远端前必须启用系统代理（`127.0.0.1:10808`）。
